# ADR-001: Вибір архітектурного стилю для сайту книжкового клубу

## Контекст
Проєкт — вебсайт книжкового клубу з можливістю:
- перегляду каталогу книг;
- реєстрації та авторизації користувачів;
- написання, редагування та перегляду відгуків;
- базової модерації відгуків.

Стан проєкту: пілотна версія (MVP).  
Команда: 1–3 розробники (full-stack).  
Процеси розробки:
- короткі ітерації;
- швидке внесення змін;
- мінімальні DevOps-процеси;
- один репозиторій;
- простий деплой на один сервер.

Очікуване навантаження — невелике, кількість користувачів обмежена.

---

## Рішення
Обрано архітектурний стиль модульний моноліт.

Застосунок розгортається як єдина система, але внутрішньо поділений на логічні модулі:
- Users (користувачі та авторизація)
- Books (каталог книг)
- Reviews (відгуки)
- Moderation (модерація контенту)

Такий підхід дозволяє швидко розробити MVP, зберігаючи чітку структуру коду.

---

## Альтернативи
### Мікросервісна архітектура
- Переваги: незалежне масштабування, технологічна гнучкість.
- Недоліки: складна інфраструктура, підвищені витрати на DevOps, надмірна складність для пілотного проєкту.

### Класичний моноліт
- Переваги: простота старту.
- Недоліки: ризик змішування бізнес-логіки та складність підтримки при зростанні функціоналу.

---

## Наслідки
### Для розробки
- простіша підтримка коду;
- чіткий поділ відповідальностей між модулями;
- легше тестування;
- швидке внесення змін.

### Для релізів
- єдиний деплой;
- мінімальні ризики під час оновлень;
- просте повернення до попередньої версії.

Можливий подальший перехід окремих модулів (наприклад, Reviews) у мікросервіси за потреби.

---

## Статус
Прийнято.
